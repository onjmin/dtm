モードの違いについて

■ SIMPLE (4 TRACKS)

シンプルな4トラック構成です。各トラックの役割があらかじめ決まっています。

  • メロディー … 曲の主役。一度に1つの音を鳴らします
  • サブメロ … メロディを支える脇役。ところどころ差し込むフレーズ向け
  • ベース … 曲の土台になる低い音。一度に1つの音を鳴らします
  • 伴奏 … コード（和音）で伴奏する役。複数の音を同時に鳴らせます

※MIDIを読み込むと、各トラックの特徴から自動でこの4つの役割に振り分けられます。

■ ADVANCED (15 TRACKS)

より自由に作り込める、15トラック構成です。

  • 最大15本のトラックを、それぞれ個別に編集できます。
  • 役割の制限や自動の振り分けはなく、どこに何の音でも自由に置けます。

※MIDIを読み込むと、ファイルのトラック構成がそのまま反映されます（1対1）。

npm: @onjmin/dtm — playMML() API解説
🚀 playMML() は、ゲームの裏側でBGMをスマートに鳴らすための「画面なし（ヘッドレス）」再生用APIです！ゲームの進行やインタラクションと音楽をぴったり同期させたいときに威力を発揮します。

■ なぜこれを使うの？（HTML5 Audio等との比較）

やりたいこと 通常のAudioタグなど @onjmin/dtm (playMML)
ズレないループ 曲の終わりと始まりの間にプチッと遅延や隙間ができがち… Web Audio APIのスケジューリングにより、ミリ秒単位で完全にシームレスなループを実現！
イントロ付きループ ファイルを分割して、再生完了を監視してつなぐ複雑な処理が必要… MML文字列を1本渡して、ループ開始位置（小節など）を指定するだけで自動ループ！
音・ビート同期演出 毎フレーム再生時間をチェックする（ズレるし、重い）… 「この小節/ステップを通ったらイベントを呼ぶ」と設定するだけで正確に発火！
音量の共通管理 ゲーム全体の音量コントロールに組み込みにくい… ゲーム側の AudioContext や MasterGain を渡せるので、SEとBGMの音量バランスもバッチリ！

■ 軽い！ロード時間がゼロのピコピコシンセ内蔵

ピアノロール画面（DAW）では音色データ（SoundFont）をロードしますが、playMML() はデフォルトで**ロード不要の超軽量シンセ（矩形波など）**で発音します。これにより、「BGMのロード待ちでゲームの起動が遅れる」「通信量がかさむ」といった問題をクリアできます。

「もっと豪華な音で鳴らしたい！」という時は、以下のように onPlayNote や onPlayDrum フックを使って、自作のシンセやゲーム側のロード済みサウンドエンジンに簡単に差し替えられます。 

playMML(mmlString, {
  audioContext: myGameCtx,
  loop: { start: { bar: 2 } }, // 2小節目からループ
  onPlayNote: (e) => {
    // 鳴らしたい音色エンジンをここに繋ぐだけ！
    mySynth.trigger(e.pitch, e.volume, e.when, e.duration);
  }
});

■ このデモで試してみよう！

  1. シームレスなイントロループ:
    ループ設定で START: BAR 2 （2小節目）にして「PLAY BGM」を押してみてください。最初は1小節目のイントロから流れますが、2周目からは自動的にイントロをスキップして2小節目からきれいにループします。
  2. リアルタイムイベント同期（Cues）:
    再生中に右側の「CONSOLE OUTPUT」を見てみてください。曲のチェックポイント（例: chorus_start など）を再生ラインが通る瞬間に、ミリ秒のズレもなくイベントログが流れます。ゲームでの演出（画面フラッシュ、キャラの動き同期など）にそのまま使えます！
